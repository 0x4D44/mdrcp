# HLD — autobin-aware binary discovery via `cargo metadata`

- **Date:** 2026-06-19
- **Repo:** `0x4D44/mdrcp` (branch `task/autobin-discovery`)
- **Status:** SIGNED-OFF (2026-06-19). Decision 1 = autobin-scan; Decision 2 = (b)
  fix the stub at its layer in deltic.
- **Author:** Claude (with Arthur)

## Problem

mdrcp deploys only the binaries it can name from **hand-parsed `Cargo.toml`
text**: `manifest_bin_names` reads `[[bin]]` entries and, when there are none,
falls back to the **package name** (`src/lib.rs:239`). It does **not** understand
Cargo **autobins** (`src/main.rs` → package-named bin, and `src/bin/*.rs`).

Consequence for deltic (which declares **no `[[bin]]`** — all four bins are
autobins): mdrcp sees only the package name `deltic`, deploys `deltic.exe`, and
silently ignores `delticw.exe` (the GUI front door) and `deltic-read.exe`, even
though `cargo build --release` produced them. That is the root cause of the
single-binary `C:\apps\deltic.exe` deployment.

## Goal / non-goals

- **Goal:** mdrcp deploys *every* binary target Cargo actually builds for the
  project (autobins included), workspace-aware, so `delticw.exe` lands in `C:\apps`.
- **Goal:** a project can keep a build-only bin (e.g. deltic's test stub
  `stub_vernier_score`) out of the deploy set.
- **Non-goal:** changing the copy/summary/Tauri/self-update behaviour. No new CLI
  flags or env vars. No back-compat shims (dev sandbox).

## Approach

Replace the text-parsing discovery with the **authoritative** Cargo target list
from `cargo metadata`, which natively resolves autobins, `[[bin]]`, workspace
members, and version inheritance — letting mdrcp delete its hand-rolled
workspace/version logic.

1. **Thin subprocess wrapper** (impure, tiny):
   `fn cargo_metadata_json(rust_base_dir) -> Result<String>` runs
   `cargo metadata --no-deps --format-version 1` with `current_dir =
   rust_base_dir` and returns stdout. `--no-deps` ⇒ `packages[]` is exactly the
   workspace members (never dependency crates). Errors with context on
   spawn-failure / non-zero exit.
2. **Pure parser** (unit-testable with JSON fixtures — no subprocess):
   `fn bin_targets_from_metadata(json) -> Result<Vec<BuiltBinary>>`:
   - for each `package` in `packages[]`: read `version`, and the opt-out list
     `package.metadata.mdrcp.exclude` (array of target names, default empty);
   - for each `target` whose `kind` array contains `"bin"` and whose name is not
     excluded: emit `BuiltBinary { base_name: target.name, version: Some(pkg.version) }`;
   - dedup by name (first writer wins — mirrors today's `HashMap` behaviour).
3. `find_built_executables` becomes: get JSON → parse → append `extra_names`
   (Tauri `productName`, paired with the root package version) → **keep the
   existing existence filter** (`target/<profile>/<name>.exe` must exist) and the
   "no executables found" error. Signature drops the now-unused `cargo_data: &Value`.
4. `run_with_options` no longer parses `Cargo.toml` into a `toml::Value` (it still
   checks the file *exists* for the friendly error). Remove the now-dead helpers
   `manifest_bin_names`, `package_version`, `expand_workspace_member` and their
   unit tests. Drop the `toml` dependency **iff** nothing else uses it.

### Stub exclusion (decision point — see D3)

deltic's `stub_vernier_score` is an autobin test fixture that should not reach
`C:\apps`. mdrcp gains the general per-package opt-out
`[package.metadata.mdrcp] exclude = ["stub_vernier_score"]`, read from cargo
metadata. deltic then adds that one line (separate 1-line change in the deltic
repo; it is `[package.metadata]`, so it does **not** alter deltic's build output
or need a version bump).

## File impact

- `src/lib.rs` — replace discovery (`find_built_executables` body + signature),
  add `cargo_metadata_json` + `bin_targets_from_metadata`, remove three dead
  helpers; trim the `Cargo.toml` parse in `run_with_options`. Swap the deleted
  helpers' unit tests for `bin_targets_from_metadata` fixture tests.
- `tests/` — one light integration test: run discovery against mdrcp's own crate
  (real `cargo metadata`) and assert `mdrcp` is found.
- `Cargo.toml` — possibly drop `toml` dep (if unused after the change).
- `README.md` / `CLAUDE.md` — one line: discovery is now via `cargo metadata`
  (all bin targets), with the `package.metadata.mdrcp.exclude` opt-out.
- **Follow-up (separate, deltic repo):** add `[package.metadata.mdrcp] exclude =
  ["stub_vernier_score"]` to `deltic/Cargo.toml`.

## Acceptance criteria

- **AC1** Running mdrcp in the deltic project (after `cargo build --release`)
  deploys `deltic.exe`, `delticw.exe`, and `deltic-read.exe`.
- **AC2** With the deltic `exclude` set, `stub_vernier_score.exe` is **not**
  deployed; without any exclude, all built bins deploy.
- **AC3** Workspace projects still deploy every member's bins (cargo metadata
  lists all `--no-deps` members).
- **AC4** Tauri projects still work: discovery runs in `src-tauri`, and
  `productName` is still added.
- **AC5** A directory with no `Cargo.toml` still fails with the existing clear
  error.
- **AC6** `cargo fmt --all` clean, `cargo clippy --all-targets -- -D warnings`
  clean, `cargo test` green.
- **AC7** `bin_targets_from_metadata` unit tests cover: bin-vs-lib filtering,
  version pairing, `exclude`, and name dedup — including a deltic-shaped fixture
  (`deltic`/`delticw`/`deltic-read`/`stub_vernier_score` with the stub excluded →
  three bins).

## Adversarial review — surviving findings (2 reviewers; verified against real `cargo metadata` + the test suite)

- **F1 (critical, verified).** Switching `run_with_options` to `cargo metadata`
  breaks mdrcp's existing integration suite: ~20 tests in `tests/integration.rs`
  build *synthetic* projects — a `Cargo.toml` (`[package]`/`[workspace]`) with
  **no `src/main.rs`** plus a touched `target/release/<n>.exe` — and call
  `run_with_options`. `cargo metadata` exits 101 ("no targets specified in the
  manifest") on those. Adopting it means adding a real `src/main.rs` to every
  synthetic test project (metadata parses sources, it does not build) and fixing
  the `test_invalid_cargo_toml` ("Failed to parse", `:74`) and empty-manifest
  assertions.
- **F2 (major).** The `[package.metadata.mdrcp].exclude` opt-out is
  overengineering — a rotting per-repo config surface invisible at the call site,
  plus cross-repo coupling. Drop it; handle the stub at its own layer or deploy-all.
- **F3 (major).** Keep the existence-filter / "no executables found" logic in a
  function that is **pure over the parsed bin list**, so it stays unit-testable;
  only a thin seam shells out (cargo-metadata option only).
- **F4 (major).** A materially simpler fix deploys `delticw.exe` with no
  subprocess and no test-suite churn: scan `src/bin/*.rs` (+ `src/bin/<name>/main.rs`,
  + the package name when `src/main.rs` exists) inside `manifest_bin_names`. The
  retained existence-filter drops anything not actually built, so the autobin
  approximation is self-correcting. (See Open decision 1.)
- **Folded correctness notes (cargo-metadata option only):** `metadata` is JSON
  `null` when `[package.metadata]` is absent (deltic's real shape) → treat as
  empty exclude; filter on `targets[].kind` containing `"bin"` (never
  `crate_types`, which is `["bin"]` for tests/build-scripts); `glob` becomes dead
  alongside `toml`.
- **Refuted / non-actionable:** duplicate bin names across members — Cargo itself
  refuses to build such a workspace, so document only; `CARGO_TARGET_DIR`
  elsewhere is a pre-existing limitation, not widened here.

## Open decisions (sign-off)

- **Decision 1 — discovery mechanism.** Given F1, I now **recommend the
  autobin-scan**: ≈10–15 lines in `manifest_bin_names` (glob `src/bin/*.rs` stems
  + `src/bin/<name>/main.rs` + package name when `src/main.rs` exists), pure, zero
  subprocess, **entire existing test suite preserved**, only a few new tests added.
  Versus `cargo metadata` (authoritative, deletes the hand-rolled workspace/version
  code — but costs the ~20-test rewrite, drops `toml`+`glob`, adds a subprocess and
  null-metadata handling). You originally chose cargo metadata; this reverses that
  on new, *verified* cost information. **Your call.**
- **Decision 2 — the deltic test stub** (`stub_vernier_score`, a real autobin, so
  it builds and would now be discovered): (a) deploy-all, accept
  `stub_vernier_score.exe` in `C:\apps` (zero extra work, mdrcp-only); or (b) stop
  it at its own layer — move `src/bin/stub_vernier_score.rs` to `examples/` or gate
  it so a release build doesn't emit it (1-line deltic change, clean `C:\apps`, no
  mdrcp config). I lean **(b)**. The dropped `metadata.mdrcp.exclude` (F2) is not
  recommended.

## Decision log

- **D1 — discovery mechanism: SUPERSEDED by Open decision 1.** Originally
  cargo-metadata (authoritative, deletes hand-rolled logic). Review finding F1
  (verified) surfaced a ~20-test rewrite cost; recommendation now flipped to the
  pure autobin-scan, pending your call.
- **D2 — wrapper + pure parser split.** Keeps the parsing logic unit-testable
  with static JSON fixtures; only a thin wrapper shells out (covered by one
  integration test).
- **D3 — stub handling: SUPERSEDED by Open decision 2.** The
  `metadata.mdrcp.exclude` opt-out is dropped (review F2: overengineering). Choose
  deploy-all or fix the stub at its own layer in deltic.
- **D4 — keep the existence filter + version pairing + copy path unchanged.** The
  filter is the safety net: anything cargo lists but didn't build is dropped.
- **D5 — `--no-deps`** so dependency crates' bins are never deployed.

## Residual risks

- **R1** `cargo metadata` needs `cargo` on PATH and a resolvable manifest; could
  touch the network if the dep graph changed. Mitigated: mdrcp runs immediately
  after a successful build, so the index is warm; failure surfaces clearly. (Not
  adding `--offline`, to match build behaviour.)
- **R2** Behaviour change fleet-wide: mdrcp now deploys *all* bin targets for
  every repo, not just the package-named one. This is the intended fix, but other
  autobin repos will start getting their extra bins in `C:\apps` (use `exclude`
  to trim). ~100–300 ms added per run (negligible).
